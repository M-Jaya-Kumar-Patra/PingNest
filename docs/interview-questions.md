# Interview Questions

## Project

Q. What is PingNest?

A.

PingNest is a developer observability platform that helps developers monitor backend applications through a reusable SDK and centralized dashboard.

---

Q. Why build PingNest?

A.

Developers repeatedly implement logging and monitoring in every project.

PingNest reduces repetitive work.

---

Q. Why SDK?

A.

Allows one-line integration.

Developers don't need to write monitoring logic repeatedly.

---

Q. Why separate SDK?

A.

SDK is an independent package.

Can be versioned and published separately.

---

Q. What is telemetry?

A.

Operational data generated by applications.

Examples:

- Requests
- Errors
- Latency

---

Q. Difference between monitoring and observability?

Monitoring

Shows problems.

Observability

Helps explain why they occurred.

---

Q. Why MongoDB?

A.

Flexible schema.

Telemetry data grows rapidly.

---

Q. Why service layer?

A.

Keeps controllers thin.

Business logic stays reusable.

---

Q. Why asyncHandler?

A.

Avoids repetitive try-catch.

Automatically forwards errors.

---

Q. Why ApiError?

A.

Creates structured errors.

---

Q. Why errorHandler?

A.

Formats every error response consistently.

---

Q. Why Helmet?

A.

Automatically adds important security headers.

---

Q. What happens if SDK crashes?

A.

Application should continue running.

SDK should fail silently and never affect the host application.

(Important design goal)

---

Q. What if the monitored app is completely down?

A.

SDK cannot send metrics.

External health checks detect downtime.


Q: Why create a logger utility instead of using console.log?

A:
Provides abstraction.
Makes future migration to Winston/Pino easier.
Keeps logging consistent.



Q: Why use Mongoose pre-save hooks?

A:
To automatically hash passwords before saving.

---

Q: What are Mongoose instance methods?

A:
Functions attached to model instances.

Example:

user.comparePassword()

---

Q: Why store JWT generation methods in model?

A:
Keeps user-related functionality inside the User entity and improves encapsulation.


Q: Why validate requests before controllers?

A:
It keeps controllers clean and ensures only valid data reaches business logic.




# Authentication Questions

Q: What is JWT?

A:
JSON Web Token is a signed token used for authentication and authorization.

---

Q: Difference between Access Token and Refresh Token?

A:

Access Token:
- Short lived
- Used for API requests

Refresh Token:
- Long lived
- Used to generate new access tokens

---

Q: Why not store JWT in localStorage?

A:

localStorage is accessible through JavaScript.

HTTP-only cookies are safer.

---

Q: Why use cookies?

A:

Cookies are automatically sent with requests and can be secured using HTTP-only flags.

---

Q: How does /me work?

A:

Login stores tokens in cookies.

Browser/Postman automatically sends cookies.

verifyJWT middleware validates token and loads user.

Controller returns req.user.

---

Q: Why use middleware for authentication?

A:

Authentication logic should be centralized instead of duplicated across controllers.

---

Q: Why hash passwords?

A:

Passwords should never be stored in plain text.

Hashing protects users if the database is compromised.

---

Q: What is req.user?

A:

User object attached by authentication middleware after successful token verification.

---

Q: Why verify user from database after decoding JWT?

A:

User may have been deleted, blocked, or had permissions changed after token creation.

What happens when the Access Token expires?
The frontend calls a refresh endpoint using the refresh token.
The backend validates the refresh token and issues a new access token.
The user remains logged in without interruption.

What happens when the Refresh Token expires?
The refresh endpoint returns 401 Unauthorized.
The frontend clears local authentication state and redirects the user to the login page.
The user must authenticate again.



Q: Why use API keys instead of user login credentials for SDK communication?

A:
SDKs are machine-to-machine integrations.

Using API keys is simpler, safer, and avoids exposing user credentials.




Q: Why use MongoDB aggregation?

A:
Aggregation performs calculations inside the database and avoids loading large datasets into application memory.


Q: Why did you create a custom SDK instead of calling the API directly?

A:
The SDK abstracts telemetry collection from application developers. Users only install the package and configure an API key. The SDK automatically measures response times, captures request metadata, and reports telemetry without requiring manual API calls.
