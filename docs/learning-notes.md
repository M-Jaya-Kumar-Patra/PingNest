# Learning Notes

# Day 1 - Project Planning

## What is PingNest?

PingNest is a developer observability platform consisting of:

- Dashboard
- Backend API
- SDK

The SDK automatically collects telemetry from Express applications and sends it to PingNest.

---

## What is Observability?

Observability is the ability to understand the internal state of an application by analyzing:

- Metrics
- Logs
- Traces

PingNest mainly focuses on Metrics.

---

## Monitoring vs Observability

Monitoring

- Checks predefined metrics.
- Example:
    - Server Up/Down
    - CPU Usage

Observability

- Helps understand WHY problems happen.

Example:

API became slow.

Observability tells:

- Which endpoint?
- Since when?
- Error rate?
- Response time?

---

## What is Telemetry?

Telemetry means operational data automatically generated by an application.

Examples:

- Request Count
- Response Time
- Error Rate
- Route Usage
- Uptime

---

## What is an SDK?

SDK = Software Development Kit

A reusable package developers install to add functionality without writing everything manually.

Example:

npm install pingnest-sdk

---

## Why Middleware?

Middleware executes before the request reaches the route.

Uses:

- Authentication
- Validation
- Logging
- Monitoring

PingNest SDK itself will be middleware.

---

## Folder Separation

PingNest has three independent products:

backend/
dashboard/
sdk/

SDK can later be published independently on npm.




Environment configuration should be centralized.

Instead of accessing process.env everywhere,
create a dedicated env configuration module.



## Mongoose Middleware

pre("save")

runs before document is saved.

Used for:

- Password hashing
- Data transformations

---

## Mongoose Instance Methods

Methods attached to schema.

Example:

user.comparePassword()

Benefits:

- Reusable
- Encapsulated
- Cleaner services


Zod allows schema-based request validation.

Validation should happen before controllers.

safeParse()

Validates data against a Zod schema.

Returns:

{
  success: true,
  data
}

or

{
  success: false,
  error
}

Unlike parse(), it does not throw exceptions.



## Mongoose Pre Hooks

Two approaches:

Callback style:

userSchema.pre("save", function(next){
    next();
});

Async style:

userSchema.pre("save", async function(){
});

Do not mix async and next() together.

In modern Mongoose, async hooks are preferred.

# Day 2 - Authentication

## JWT Authentication Flow

1. User logs in.
2. Server verifies credentials.
3. Server generates:
   - Access Token
   - Refresh Token
4. Tokens are stored in cookies.
5. Browser/Postman automatically sends cookies.
6. Middleware verifies access token.
7. User is authenticated.

---

## Access Token

Purpose:
Authenticate API requests.

Lifetime:
Short (15 minutes).

Used on:
Every protected request.

Example:

GET /api/v1/auth/me

---

## Refresh Token

Purpose:
Generate new access tokens.

Lifetime:
Long (7 days).

Stored:
Database + Cookie

---

## HTTP-only Cookies

JavaScript cannot access them.

Benefits:
- Better security
- Protection against token theft

Example:

res.cookie(
  "accessToken",
  token,
  cookieOptions
)

---

## Protected Routes

Protected routes require authentication.

Example:

GET /api/v1/auth/me

Flow:

Request
 ↓
verifyJWT
 ↓
Controller

---

## Middleware

Middleware runs before controller execution.

Examples:

- Authentication
- Validation
- Logging

---

## Postman Cookie Storage

Postman automatically stores cookies received through:

Set-Cookie header

and sends them on future requests.

This is why /me worked without manually sending cookies.

## Error vs ApiError

Use ApiError for request-related failures:

- Unauthorized
- Validation errors
- Resource not found

Use Error for application/programming failures:

- Missing environment variables
- Service initialization failures
- Socket.IO not initialized
- Unexpected internal bugs

ApiError represents expected operational errors.

Error represents unexpected system/programmer errors.
